Основні команди для перегляду логів:

journalctl: Показує всі системні логи.
journalctl -u service_name: Логи конкретного сервісу (наприклад, nginx).
journalctl -f: Перегляд нових логів у реальному часі.

ip addr (або коротко ip a): Перевірити IP-адреси на всіх інтерфейсах. Тут ми дивимося, чи піднятий інтерфейс (UP) і яка в нього маска.

ip link: Стан фізичного рівня. Дозволяє побачити MAC-адреси та чи взагалі система бачить "кабель".

logrotate — механізм, який не дає логам "з'їсти" все місце на диску сервера

змінили конфіг і хочете перевірити його:

Роботи з дисками
Категорія,    Розуміти (Логіка),                                              Вивчити (Команди)
Disk Space,   Видалення файлу не завжди звільняє місце миттєво.,                    "df -h, du -sh"
Inodes,       "Файлів може бути "багато", навіть якщо вони "маленькі",               df -i
I/O Wait,     Процесор чекає диск = диск забитий або повільний.,                     "iotop, iostat"
Permissions,  "Чому root може все, а твій юзер — ні.",                               "chmod, chown"
 
Сценарій: Твій додаток генерує мільйони крихітних лог-файлів або сесій. Місце на диску є, але ліміт інод вичерпано. Система більше не може створити жодного нового файлу.

df -i — Це перевірка заповненості інод (не місця, а саме кількості файлів)
Якщо df -i показав 100%, використай цю команду, щоб знайти папку, де найбільше файлів:
find / -xdev -type d -exec sh -c 'echo "$(ls -a "$1" | wc -l) $1"' _ {} \; | sort -rn | head -n 10

iotop — Аналог top, але для диска. Показує, який саме процес зараз "ґвалтує" диск записом чи читанням.
du -sh /var/log/* — Швидкий спосіб дізнатися, яка папка "з'їла" все місце.
lsof (List Open Files) — Дозволяє побачити, які файли зараз відкриті процесами




#Docker
Download Docker Desktop for WSL
Збірка образу:
docker build -t my-first-app .
(Крапка в кінці означає "шукай Dockerfile у цій папці")

Зупини та видали його (однією командою):
docker rm -f <ID_або_ім'я>

Запуск контейнера:
docker run -d -p 8080:8000 my-first-app

-d: Запустити у фоні (detached).

-p 8080:8000: Прокинути порт. Зовнішній порт 8080 (твій ПК) веде на внутрішній 8000 (контейнер).

Команди для очищення системи

Коли ти багато експериментуєш, Docker з'їдає диск (пам'ятаєш про df -h?).
docker ps -a — показати всі контейнери (навіть зупинені).
docker rm <ID> — видалити контейнер.
docker images — список образів.
docker system prune — МАГІЧНА КОМАНДА. Видаляє всі зупинені контейнери та невикористані шари, звільняючи купу місця.

Команди для керування docker-compose.yml

docker-compose up -d — запустити всю "зграю" контейнерів у фоні.
docker-compose ps — подивитися статус усіх сервісів проекту.
docker-compose stop — зупинити все.
docker-compose logs -f web — дивитися "живі" логи тільки веб-сервера.

Збірка образу (Build)
Оскільки ми змінили структуру Dockerfile, нам треба перезібрати образ, щоб Docker створив нові шари:
docker build -t pro-app .
docker run -d -p 8080:8000 --name my-pro-container pro-app

як перевірити, чи потрапив секретний файл .env всередину контейнера?

docker exec -it <container_id_or_name> sh (або bash)

-i (interactive) — тримати вхід відкритим.
-t (tty) — виділити віртуальний термінал (щоб це виглядало як звичайна командна строка).
sh — запустити оболонку всередині контейнера.
Коли ти "всередині", ти можеш просто ввести ls -la і побачити, чи є там файл .env.


docker exec -it devops-web-1 whoami
причину смерті контейнера:
docker logs devops-web-1

Перезберіть та запустіть:
Оскільки ми змінили код (app.py), треба оновити образ:
docker-compose up -d --build
(Прапор --build змушує Docker перезібрати шари, де копіюється код)

мережевий міст Docker "зависає"? 
sudo service docker restart
